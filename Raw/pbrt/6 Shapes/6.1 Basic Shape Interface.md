## 6.1 Basic Shape Interface

The interface for Shapes is defined in the file base/shape.h, and the shape implementations can be found in shapes.h and shapes.cpp. The Shape class defines the general shape interface.

> Shape 的接口定义在 base/shape.h 文件中，形状的实现可以在 shapes.h 和 shapes.cpp 中找到。Shape 类定义了通用的形状接口。

Shape is a TaggedPointer that can hold Sphere, Cylinder, Disk, Triangle, BilinearPatch, or Curve. It provides methods for Bounds, NormalBounds, Intersect, IntersectP, Area, Sample, and PDF.

> Shape 是一个 TaggedPointer，可以持有 Sphere、Cylinder、Disk、Triangle、BilinearPatch 或 Curve。它提供了 Bounds、NormalBounds、Intersect、IntersectP、Area、Sample 和 PDF 等方法。

### 6.1.1 Bounding

The scenes that pbrt renders often contain objects that are computationally expensive to process. For many operations, it is useful to have a 3D *bounding volume* that encloses an object. For example, if a ray does not pass through a particular bounding volume, pbrt can avoid processing all the objects inside of it for that ray.

> pbrt 渲染的场景通常包含计算代价高昂的对象。对于许多操作，拥有一个包围对象的三维*包围体*是很有用的。例如，如果光线没有穿过某个特定的包围体，pbrt 就可以避免为该光线处理其中的所有对象。

Axis-aligned bounding boxes are a convenient bounding volume, as they require only six floating-point values to store. They fit many shapes well and it is fairly inexpensive to test for the intersection of a ray with an axis-aligned bounding box. Each Shape implementation must therefore be capable of bounding itself with an axis-aligned bounding box represented by a Bounds3f. The returned bounding box should be in the rendering coordinate system.

> 轴对齐包围盒是一种方便的包围体，因为它们只需要六个浮点值来存储。它们能很好地拟合许多形状，而且测试光线与轴对齐包围盒的交叉相对廉价。因此，每个 Shape 实现都必须能够用 Bounds3f 表示的轴对齐包围盒来包围自身。返回的包围盒应在渲染坐标系中。

In addition to bounding their spatial extent, shapes must also be able to bound their range of surface normals. The NormalBounds() method should return such a bound using a DirectionCone, which was defined in Section 3.8.4. Normal bounds are specifically useful in lighting calculations: when a shape is emissive, they sometimes make it possible to efficiently determine that the shape does not illuminate a particular point in the scene.

> 除了包围空间范围外，形状还必须能够界定其表面法线的范围。NormalBounds() 方法应使用 DirectionCone（在 §3.8.4 中定义）返回这样的界。法线界在光照计算中特别有用：当形状是发光体时，它们有时可以高效地判定该形状不会照亮场景中的某个特定点。

### 6.1.2 Ray–Bounds Intersections

Given the use of Bounds3f instances to bound shapes, we will add a Bounds3 method, Bounds3::IntersectP(), that checks for a ray–box intersection and returns the two parametric values of the intersection, if any.

> 鉴于使用 Bounds3f 实例来包围形状，我们将添加一个 Bounds3 方法 Bounds3::IntersectP()，它检测光线-盒体交叉并返回交叉点的两个参数值（如果存在）。

One way to think of bounding boxes is as the intersection of three slabs, where a slab is the region of space between two parallel planes. To intersect a ray with a box, we intersect the ray with each of the box's three slabs in turn. Because the slabs are aligned with the three coordinate axes, a number of optimizations can be made in the ray–slab tests.

> 包围盒的一种理解方式是将其视为三个平板（slab）的交集，其中平板是两个平行平面之间的空间区域。为了将光线与盒体求交，我们依次将光线与盒体的三个平板求交。由于平板与三个坐标轴对齐，可以在光线-平板测试中进行多种优化。

The basic ray–bounding box intersection algorithm works as follows: we start with a parametric interval that covers that range of positions along the ray where we are interested in finding intersections; typically, this is (0, tMax). We will then successively compute the two parametric positions where the ray intersects each axis-aligned slab. We compute the set intersection of the per-slab intersection interval with the current intersection interval, returning failure if we find that the resulting interval is degenerate. If, after checking all three slabs, the interval is nondegenerate, we have the parametric range of the ray that is inside the box.

> 基本的光线-包围盒求交算法如下：我们从一个参数区间开始，该区间覆盖了我们关注的沿光线的位置范围；通常为 (0, tMax)。然后我们依次计算光线与每个轴对齐平板相交的两个参数位置。我们计算每个平板交叉区间与当前交叉区间的集合交集，如果发现所得区间退化（为空）则返回失败。如果检查完所有三个平板后区间非退化，我们就得到了光线在盒体内部的参数范围。

If the Bounds3::IntersectP() method returns true, the intersection's parametric range is returned in the optional arguments hitt0 and hitt1. Intersections outside of the (0, tMax) range of the ray are ignored. If the ray's origin is inside the box, 0 is returned for hitt0.

> 如果 Bounds3::IntersectP() 方法返回 true，交叉的参数范围通过可选参数 hitt0 和 hitt1 返回。光线 (0, tMax) 范围之外的交叉被忽略。如果光线原点在盒体内部，hitt0 返回 0。

For each pair of planes, this routine needs to compute two ray–plane intersections. For a plane perpendicular to the x axis, the parametric t value simplifies to: t = (pPlane.x - o.x) / d.x. The code computes the reciprocal of the direction component so that it can multiply instead of dividing. Note that it is not necessary to verify that the direction component is nonzero—if it is zero, invRayDir will hold an infinite value, and the rest of the algorithm still works correctly.

> 对于每对平面，该例程需要计算两个光线-平面交叉。对于垂直于 x 轴的平面，参数 t 值简化为：t = (pPlane.x - o.x) / d.x。代码计算方向分量的倒数，以便用乘法代替除法。注意无需验证方向分量是否为零——如果为零，invRayDir 将持有一个无穷值，而算法的其余部分仍然正确工作。

The two distances are reordered so that tNear holds the closer intersection and tFar the farther one. This gives a parametric range, which is used to compute the set intersection with the current range to compute a new range. If this new range is empty (i.e., t0 > t1), then the code can immediately return failure.

> 两个距离被重新排序，使 tNear 持有较近的交叉而 tFar 持有较远的交叉。这给出一个参数范围，用于与当前范围计算集合交集以得到新范围。如果新范围为空（即 t0 > t1），则代码可以立即返回失败。

There is another floating-point-related subtlety here: in the case where the ray origin is in the plane of one of the bounding box slabs and the ray lies in the plane of the slab, it is possible that tNear or tFar will be computed by an expression of the form 0/0, which results in a floating-point "not a number" (NaN) value. Like infinity values, NaNs have well-specified semantics: any logical comparison involving a NaN always evaluates to false. Therefore, the code that updates the values of t0 and t1 is carefully written so that if tNear or tFar is NaN, then t0 or t1 will not ever take on a NaN value but will always remain unchanged.

> 这里还有一个与浮点数相关的微妙之处：当光线原点位于包围盒某个平板的平面上且光线位于该平板平面内时，tNear 或 tFar 可能由 0/0 形式的表达式计算，这会产生浮点"非数字"（NaN）值。与无穷值类似，NaN 有明确的语义规范：涉及 NaN 的任何逻辑比较始终求值为 false。因此，更新 t0 和 t1 值的代码经过精心编写，使得如果 tNear 或 tFar 是 NaN，则 t0 或 t1 永远不会变为 NaN 值，而是始终保持不变。

Bounds3 also provides a specialized IntersectP() method that takes the reciprocal of the ray's direction as an additional parameter, so that the three reciprocals do not need to be computed each time IntersectP() is called.

> Bounds3 还提供了一个特化的 IntersectP() 方法，它将光线方向的倒数作为额外参数，这样三个倒数就不必在每次调用 IntersectP() 时重新计算。

This version of the method also takes precomputed values that indicate whether each direction component is negative, which makes it possible to eliminate the comparisons of the computed tNear and tFar values in the original routine and to directly compute the respective near and far values. Because the comparisons that order these values from low to high in the original code are dependent on computed values, they can be inefficient for processors to execute, since the computation of their values must be finished before the comparison can be made. Because many ray–bounds intersection tests may be performed during rendering, this small optimization is worth using.

> 该方法的这个版本还接收预计算的值，指示每个方向分量是否为负，这使得可以消除原始例程中对计算出的 tNear 和 tFar 值的比较，直接计算相应的近值和远值。由于原始代码中排序这些值的比较依赖于计算值，对处理器来说执行效率可能较低，因为必须等待值计算完成才能进行比较。由于渲染期间可能执行大量光线-包围盒求交测试，这个小优化值得使用。

This intersection test is at the heart of traversing the BVHAggregate acceleration structure, which is introduced in Section 7.3. Because so many ray–bounding box intersection tests are performed while traversing the BVH tree, we found that this optimized method provided approximately a 15% performance improvement in overall rendering time compared to using the Bounds3::IntersectP() variant that did not take the precomputed direction reciprocals and signs.

> 这个求交测试是遍历 BVHAggregate 加速结构（将在 §7.3 中介绍）的核心。由于遍历 BVH 树时会执行大量光线-包围盒求交测试，我们发现这个优化方法与不使用预计算方向倒数和符号的 Bounds3::IntersectP() 变体相比，在总渲染时间上提供了约 15% 的性能提升。

### 6.1.3 Intersection Tests

Shape implementations must provide an implementation of two methods that test for ray intersections with their shape. The first, Intersect(), returns geometric information about a single ray–shape intersection corresponding to the first intersection, if any, in the (0, tMax) parametric range along the given ray.

> Shape 实现必须提供两个方法来测试光线与其形状的交叉。第一个方法 Intersect() 返回关于单个光线-形状交叉的几何信息，对应于给定光线沿 (0, tMax) 参数范围内的第一个交叉点（如果存在）。

In the event that an intersection is found, a SurfaceInteraction corresponding to the intersection point and the parametric distance along the ray where the intersection occurred are returned via a ShapeIntersection instance.

> 如果找到交叉，则通过 ShapeIntersection 实例返回对应于交叉点的 SurfaceInteraction 和交叉发生处沿光线的参数距离。

There are a few important things to keep in mind when reading (and writing) intersection routines:

> 在阅读（和编写）求交例程时，有几个重要事项需要记住：

- The provided tMax value defines the endpoint of the ray. Intersection routines must ignore any intersections that occur after this point.
- If there are multiple intersections with a shape along the ray, the closest one should be reported.
- The rays passed into intersection routines are in rendering space, so shapes are responsible for transforming them to object space if needed for intersection tests. The intersection information returned should be in rendering space.

> - 提供的 tMax 值定义了光线的端点。求交例程必须忽略在此点之后发生的任何交叉。
> - 如果沿光线与形状有多个交叉，应报告最近的那个。
> - 传入求交例程的光线处于渲染空间，因此形状负责在需要时将其变换到物体空间进行求交测试。返回的交叉信息应在渲染空间中。

The second intersection test method, Shape::IntersectP(), is a predicate function that determines whether or not an intersection occurs without returning any details about the intersection itself. That test is often more efficient than a full intersection test. This method is used in particular for shadow rays that are testing the visibility of a light source from a point in the scene.

> 第二个求交测试方法 Shape::IntersectP() 是一个谓词函数，它判定交叉是否发生而不返回关于交叉本身的任何细节。该测试通常比完整的求交测试更高效。此方法特别用于阴影光线，测试从场景中某点到光源的可见性。

### 6.1.4 Intersection Coordinate Spaces

For some shapes, intersection routines are most naturally expressed in their object space. For example, the Sphere shape computes the intersection with a sphere of a given radius positioned at the origin. Other shapes, like the Triangle, transform their representation to rendering space and perform intersection tests there.

> 对于某些形状，求交例程最自然地在其物体空间中表达。例如，Sphere 形状计算与位于原点的给定半径球体的交叉。其他形状如 Triangle，则将其表示变换到渲染空间并在那里执行求交测试。

Shapes like Sphere that operate in object space must transform the specified ray to object space and then transform any intersection results back to rendering space. A natural question is: what effect does the object-from-rendering-space transformation have on the correct parametric distance to return?

> 像 Sphere 这样在物体空间中操作的形状必须将指定的光线变换到物体空间，然后将任何交叉结果变换回渲染空间。一个自然的问题是：从渲染空间到物体空间的变换对需要返回的正确参数距离有什么影响？

Using the properties of transformations, it is possible to show that the distance to the intersection is unaffected by the transformation. If the ray–shape intersection algorithm finds an object-space intersection at a distance t along the ray, then the rendering-space intersection point is found by applying the inverse transformation. Therefore, the t value computed in object space is the correct value for the intersection point in rendering space as well. Note that if the object-space ray's direction had been normalized after the transformation, then this would no longer be the case. This is one reason that pbrt does not normalize object-space rays' directions after transformation.

> 利用变换的性质，可以证明到交叉点的距离不受变换影响。如果光线-形状求交算法在物体空间中以沿光线距离 t 找到交叉，那么渲染空间中的交叉点可通过应用逆变换得到。因此，在物体空间中计算的 t 值也是渲染空间中交叉点的正确值。注意，如果物体空间光线的方向在变换后被归一化了，则这不再成立。这就是 pbrt 在变换后不归一化物体空间光线方向的原因之一。

### 6.1.5 Sidedness

Many rendering systems, particularly those based on scanline or z-buffer algorithms, support the concept of shapes being "one-sided"—the shape is visible if seen from the front but disappears when viewed from behind. The potential for improved performance is reduced when using this technique with ray tracing, however, since it is often necessary to perform the ray–object intersection before determining the surface normal to do the backfacing test. Furthermore, this feature can lead to a physically inconsistent scene description if one-sided objects are not in fact closed. For all of these reasons, pbrt does not support this feature.

> 许多渲染系统，特别是基于扫描线或 z 缓冲算法的系统，支持形状"单面"的概念——形状从正面看是可见的，但从背面看则消失。然而，将此技术用于光线追踪时性能提升的潜力有限，因为通常需要先执行光线-物体求交才能确定表面法线以进行背面测试。此外，如果单面对象实际上不是封闭的，此特性可能导致物理上不一致的场景描述。基于所有这些原因，pbrt 不支持此特性。

### 6.1.6 Area

In pbrt, area lights are defined by attaching an emission profile to a Shape. To use Shapes as area lights, it is necessary that shapes be able to return their surface area in rendering space.

> 在 pbrt 中，面光源通过将发射分布附加到 Shape 来定义。为了将 Shape 用作面光源，形状必须能够返回其在渲染空间中的表面积。

### 6.1.7 Sampling

A few methods are necessary to sample points on the surface of shapes in order to use them as emitters. There are two shape sampling methods, both named Sample().

> 为了将形状用作发射体，需要几个方法来对形状表面上的点进行采样。有两个形状采样方法，都名为 Sample()。

The first chooses points on the surface of the shape using a sampling distribution with respect to surface area and returns the local geometric information about the sampled point in a ShapeSample. The provided sample value u, a uniform sample in [0,1)², should be used to determine the point on the shape.

> 第一个方法使用相对于表面积的采样分布在形状表面上选择点，并在 ShapeSample 中返回采样点的局部几何信息。提供的采样值 u 是 [0,1)² 中的均匀样本，应用于确定形状上的点。

The ShapeSample structure stores an Interaction corresponding to a sampled point on the surface as well as the probability density with respect to surface area on the shape for sampling that point.

> ShapeSample 结构存储对应于表面上采样点的 Interaction，以及在该形状上对该点采样的相对于表面积的概率密度。

Shapes must also provide an associated PDF() method that returns probability density for sampling the specified point on the shape that corresponds to the given Interaction. This method should only be called with interactions that are on the shape's surface. Although Sample() already returns the probability density for the point it samples, this method is useful when using multiple importance sampling, in which case it is necessary to compute the probability density for samples generated using other sampling techniques.

> Shape 还必须提供一个关联的 PDF() 方法，返回对形状上与给定 Interaction 对应的指定点进行采样的概率密度。此方法应仅使用位于形状表面上的 interaction 调用。虽然 Sample() 已经返回了其采样点的概率密度，但此方法在使用多重要性采样时很有用，在这种情况下需要计算由其他采样技术生成的样本的概率密度。

The second shape sampling method takes a reference point from which the shape is being viewed. This method is particularly useful for lighting, since the caller can pass in the point to be lit and allow shape implementations to ensure that they only sample the portion of the shape that is potentially visible from that point.

> 第二个形状采样方法接收一个观察形状的参考点。此方法对于光照特别有用，因为调用者可以传入需要被照亮的点，并允许形状实现确保它们只采样从该点可能可见的形状部分。

Unlike the first Shape sampling method, which generates points on the shape according to a probability density with respect to surface area on the shape, the second one uses a density with respect to solid angle from the reference point. This difference stems from the fact that the area light sampling routines evaluate the direct lighting integral as an integral over directions from the reference point—expressing these sampling densities with respect to solid angle at the point is more convenient.

> 与第一个按相对于形状表面积的概率密度在形状上生成点的 Shape 采样方法不同，第二个方法使用相对于从参考点出发的立体角的密度。这一差异源于面光源采样例程将直接光照积分表示为从参考点出发的方向上的积分——用该点处的立体角来表达这些采样密度更为方便。

Information about the reference point and its geometric and shading normals is provided by the ShapeSampleContext structure. The reference point position is specified using the Point3fi class, which can represent the numerical uncertainty in a ray intersection point computed using floating-point arithmetic. For points in participating media that are not associated with a surface, the normal and shading normal are left with their default values of (0,0,0).

> 关于参考点及其几何法线和着色法线的信息由 ShapeSampleContext 结构提供。参考点位置使用 Point3fi 类指定，该类可以表示使用浮点算术计算的光线交叉点中的数值不确定性。对于不与表面关联的参与介质中的点，法线和着色法线保留其默认值 (0,0,0)。

A second PDF() method comes along with this sampling approach. It returns the shape's probability of sampling a point on the light such that the incident direction at the reference point is wi. As with the corresponding sampling method, this density should be with respect to solid angle at the reference point.

> 第二个 PDF() 方法伴随这种采样方法。它返回形状对光源上某点的采样概率，使得在参考点处的入射方向为 wi。与对应的采样方法一样，该密度应相对于参考点处的立体角。

Some of the PDF() method implementations will need to trace a ray from the reference point in the direction wi to see if it intersects the shape. The ShapeSampleContext methods OffsetRayOrigin() and SpawnRay() should be used to find the origin or the ray itself rather than using the point returned by ShapeSampleContext::p(). This stems from a subtlety related to handling numeric error in intersection points; the implementation of these methods and discussion of the underlying issues can be found in Section 6.8.6.

> 一些 PDF() 方法实现需要从参考点沿 wi 方向追踪光线以查看它是否与形状相交。应使用 ShapeSampleContext 的 OffsetRayOrigin() 和 SpawnRay() 方法来获取光线原点或光线本身，而不是使用 ShapeSampleContext::p() 返回的点。这源于处理交叉点数值误差的微妙之处；这些方法的实现和底层问题的讨论可以在 §6.8.6 中找到。
