## 1.1 Literate Programming

> ## 1.1 文学编程

While creating the T <sub>E</sub> X typesetting system, Donald Knuth developed a new programming methodology based on a simple but revolutionary idea. To quote Knuth, "let us change our traditional attitude to the construction of programs: Instead of imagining that our main task is to instruct a computer what to do, let us concentrate rather on explaining to human beings what we want a computer to do." He named this methodology *literate programming*. This book (including the chapter you are reading now) is a long literate program. This means that in the course of reading this book, you will read the full implementation of the pbrt rendering system, not just a high-level description of it.

> 在创建 T<sub>E</sub>X 排版系统时，Donald Knuth 基于一个简单但革命性的想法开发了一种新的编程方法论。引用 Knuth 的话："让我们改变对程序构建的传统态度：不要把我们的主要任务想象为指示计算机做什么，而是让我们专注于向人类解释我们想让计算机做什么。"他将这种方法论命名为*文学编程*（literate programming）。本书（包括你正在阅读的这一章）就是一个长篇文学程序。这意味着在阅读本书的过程中，你将阅读到 pbrt 渲染系统的完整实现，而不仅仅是对它的高层描述。

Literate programs are written in a metalanguage that mixes a document formatting language (e.g., T <sub>E</sub> X or HTML) and a programming language (e.g., C++). Two separate systems process the program: a "weaver" that transforms the literate program into a document suitable for typesetting and a "tangler" that produces source code suitable for compilation. Our literate programming system is homegrown, but it was heavily influenced by Norman Ramsey's noweb system.

> 文学程序用一种元语言编写，该元语言混合了文档格式化语言（如 T<sub>E</sub>X 或 HTML）和编程语言（如 C++）。两个独立的系统处理该程序：一个"编织器"（weaver）将文学程序转换为适合排版的文档，一个"纠缠器"（tangler）生成适合编译的源代码。我们的文学编程系统是自行开发的，但深受 Norman Ramsey 的 noweb 系统的影响。

The literate programming metalanguage provides two important features. The first is the ability to mix prose with source code. This feature puts the description of the program on equal footing with its actual source code, encouraging careful design and documentation. Second, the language provides mechanisms for presenting the program code to the reader in an order that is entirely different from the compiler input. Thus, the program can be described in a logical manner. Each named block of code is called a *fragment*, and each fragment can refer to other fragments by name.

> 文学编程元语言提供了两个重要特性。第一个是将散文与源代码混合的能力。这一特性将程序的描述与其实际源代码置于同等地位，鼓励精心的设计和文档化。第二，该语言提供了以与编译器输入完全不同的顺序向读者呈现程序代码的机制。因此，程序可以以合乎逻辑的方式来描述。每个命名的代码块称为一个*片段*（fragment），每个片段可以通过名称引用其他片段。

As a simple example, consider a function InitGlobals() that is responsible for initializing all of a program's global variables:

> 作为一个简单的例子，考虑一个负责初始化程序所有全局变量的函数 InitGlobals()：

void InitGlobals() { nMarbles = 25.7; shoeSize = 13; dielectric = true; }

Despite its brevity, this function is hard to understand without any context. Why, for example, can the variable nMarbles take on floating-point values? Just looking at the code, one would need to search through the entire program to see where each variable is declared and how it is used in order to understand its purpose and the meanings of its legal values. Although this structuring of the system is fine for a compiler, a human reader would much rather see the initialization code for each variable presented separately, near the code that declares and uses the variable.

> 尽管很简短，但在没有任何上下文的情况下，这个函数很难理解。例如，为什么变量 nMarbles 可以取浮点值？仅看代码，人们需要在整个程序中搜索每个变量的声明位置和使用方式，才能理解其目的和合法值的含义。虽然这种系统结构对编译器来说没问题，但人类读者更希望看到每个变量的初始化代码单独呈现，并且靠近声明和使用该变量的代码。

In a literate program, one can instead write InitGlobals() like this:

> 在文学程序中，可以像这样编写 InitGlobals()：

<<Function Definitions>>=

void InitGlobals() { << [Initialize Global Variables](#fragment-InitializeGlobalVariables-0) >>

shoeSize = 13; dielectric = true;

}

This defines a fragment, called << [Function Definitions](#fragment-FunctionDefinitions-0) >>, that contains the definition of the InitGlobals() function. The InitGlobals() function itself refers to another fragment, << [Initialize Global Variables](#fragment-InitializeGlobalVariables-0) >>. Because the initialization fragment has not yet been defined, we do not know anything about this function except that it will presumably contain assignments to global variables. (However, we can peek ahead by clicking on the plus sign on the right side of it; doing so expands out all the fragment's final code.)

> 这定义了一个名为 << [Function Definitions](#fragment-FunctionDefinitions-0) >> 的片段，其中包含 InitGlobals() 函数的定义。InitGlobals() 函数本身引用了另一个片段 << [Initialize Global Variables](#fragment-InitializeGlobalVariables-0) >>。由于初始化片段尚未定义，我们对这个函数一无所知，只知道它大概会包含对全局变量的赋值。（不过，我们可以通过点击右侧的加号来提前查看；这样做会展开片段的所有最终代码。）

Just having the fragment name is just the right level of abstraction for now, since no variables have been declared yet. When we introduce the global variable shoeSize somewhere later in the program, we can then write

> 目前仅有片段名称就是恰当的抽象层次，因为还没有声明任何变量。当我们在程序后面某处引入全局变量 shoeSize 时，我们可以写

<<Initialize Global Variables>>=

shoeSize = 13;

Here we have started to define the contents of << [Initialize Global Variables](#fragment-InitializeGlobalVariables-0) >>. When the literate program is tangled into source code for compilation, the literate programming system will substitute the code shoeSize = 13; inside the definition of the InitGlobals() function. The symbol after the equals sign indicates that more code will later be added to this fragment. Clicking on it brings you to where that happens.

> 在这里我们开始定义 << [Initialize Global Variables](#fragment-InitializeGlobalVariables-0) >> 的内容。当文学程序被"纠缠"成用于编译的源代码时，文学编程系统会将代码 `shoeSize = 13;` 替换到 InitGlobals() 函数的定义中。等号后面的符号表示稍后会有更多代码添加到这个片段中。点击它可以跳转到添加代码的位置。

Later in the text, we may define another global variable, dielectric, and we can append its initialization to the fragment:

> 在文本的后面部分，我们可能会定义另一个全局变量 dielectric，我们可以将其初始化追加到片段中：

<<Initialize Global Variables>>+=

dielectric = true;

The += symbol after the fragment name shows that we have added to a previously defined fragment. Further, the symbol links back to the previous place where << [Initialize Global Variables](#fragment-InitializeGlobalVariables-0) >> had code added to it.

> 片段名称后的 `+=` 符号表示我们已向先前定义的片段追加了内容。此外，该符号链接回 << [Initialize Global Variables](#fragment-InitializeGlobalVariables-0) >> 上次添加代码的位置。

When tangled, these three fragments turn into the code

> 当被"纠缠"后，这三个片段变成以下代码

void InitGlobals() { // Initialize Global Variables shoeSize = 13; dielectric = true; }

In this way, we can decompose complex functions into logically distinct parts, making them much easier to understand. For example, we can write a complicated function as a series of fragments:

> 通过这种方式，我们可以将复杂函数分解为逻辑上独立的部分，使它们更容易理解。例如，我们可以将一个复杂函数写成一系列片段：

<<Function Definitions>>+=

void complexFunc(int x, int y, double \*values) { <<Check validity of arguments>> if (x < y) { <<Swap x and y >> } <<Do precomputation before loop>> <<Loop through and update values array>> }

Again, the contents of each fragment are expanded inline in complexFunc() for compilation. In the document, we can introduce each fragment and its implementation in turn. This decomposition lets us present code a few lines at a time, making it easier to understand. Another advantage of this style of programming is that by separating the function into logical fragments, each with a single and well-delineated purpose, each one can then be written, verified, or read independently. In general, we will try to make each fragment less than 10 lines long.

> 同样，每个片段的内容在编译时会被内联展开到 complexFunc() 中。在文档中，我们可以依次介绍每个片段及其实现。这种分解方式让我们可以每次只呈现几行代码，使其更容易理解。这种编程风格的另一个优势是，通过将函数分离为逻辑片段，每个片段都有单一且明确的目的，每个片段可以独立编写、验证或阅读。通常，我们会尽量使每个片段不超过 10 行。

In some sense, the literate programming system is just an enhanced macro substitution package tuned to the task of rearranging program source code. This may seem like a trivial change, but in fact literate programming is quite different from other ways of structuring software systems.

> 从某种意义上说，文学编程系统只是一个经过增强的宏替换包，专门用于重新组织程序源代码。这看起来可能是一个微不足道的变化，但实际上文学编程与其他构建软件系统的方式有很大不同。
