## 1.3 pbrt: System Overview

pbrt is structured using standard object-oriented techniques: for each of a number of fundamental types, the system specifies an interface that implementations of that type must fulfill. For example, pbrt requires the implementation of a particular shape that represents geometry in a scene to provide a set of methods including one that returns the shape's bounding box, and another that tests for intersection with a given ray. In turn, the majority of the system can be implemented purely in terms of those interfaces; for example, the code that checks for occluding objects between a light source and a point being shaded calls the shape intersection methods without needing to consider which particular types of shapes are present in the scene.

> pbrt 使用标准的面向对象技术构建：对于若干基本类型，系统指定了该类型的实现必须满足的接口。例如，pbrt 要求表示场景中几何体的特定形状实现提供一组方法，包括一个返回形状包围盒的方法和一个测试与给定光线是否相交的方法。反过来，系统的大部分可以纯粹基于这些接口来实现；例如，检查光源和着色点之间是否有遮挡物体的代码调用形状求交方法，而无需考虑场景中存在哪些特定类型的形状。

There are a total of 14 of these key base types. Adding a new implementation of one of these types to the system is straightforward; the implementation must provide the required methods, it must be compiled and linked into the executable, and the scene object creation routines must be modified to create instances of the object as needed as the scene description file is parsed.

> 共有 14 种这样的关键基类型。向系统添加这些类型之一的新实现很简单；该实现必须提供所需的方法，必须编译并链接到可执行文件中，并且必须修改场景对象创建例程，以便在解析场景描述文件时根据需要创建对象实例。

Conventional practice in C++ would be to specify the interfaces for each of these types using abstract base classes that define pure virtual functions and to have implementations inherit from those base classes and implement the required virtual functions. That approach was used in the three previous versions of pbrt, but the addition of support for rendering on graphics processing units (GPUs) in this version motivated a more portable approach based on *tag-based dispatch*, where each specific type implementation is assigned a unique integer that determines its type at runtime.

> C++ 中的传统做法是使用定义纯虚函数的抽象基类来指定每种类型的接口，并让实现继承这些基类并实现所需的虚函数。这种方法在 pbrt 的前三个版本中使用，但本版本增加了对 GPU 渲染的支持，促使采用一种基于*标签分派*（tag-based dispatch）的更具可移植性的方法，其中每个特定类型的实现被赋予一个唯一的整数，在运行时确定其类型。

This version of pbrt is capable of running on GPUs that support C++17 and provide APIs for ray intersection tests. We have carefully designed the system so that almost all of pbrt's implementation runs on both CPUs and GPUs, just as it is presented in Chapters 2 through 12. We will therefore generally say little about the CPU versus the GPU in most of the following.

> 本版本的 pbrt 能够在支持 C++17 并提供光线求交测试 API 的 GPU 上运行。我们精心设计了系统，使得几乎所有 pbrt 的实现都能在 CPU 和 GPU 上运行，正如第 2 至第 12 章所呈现的那样。因此，在以下大部分内容中，我们一般很少提及 CPU 与 GPU 的区别。

The main differences between the CPU and GPU rendering paths in pbrt are in their data flow and how they are parallelized—effectively, how the pieces are connected together. Both the basic rendering algorithm described later in this chapter and the light transport algorithms described in Chapters 13 and 14 are only available on the CPU. The GPU rendering pipeline is discussed in Chapter 15, though it, too, is also capable of running on the CPU.

> pbrt 中 CPU 和 GPU 渲染路径的主要区别在于它们的数据流和并行化方式——实际上是各部分如何连接在一起。本章稍后描述的基本渲染算法和第 13、14 章描述的光传输算法仅在 CPU 上可用。GPU 渲染管线在第 15 章讨论，不过它也能在 CPU 上运行。

### 1.3.1 Phases of Execution

pbrt can be conceptually divided into three phases of execution. First, it parses the scene description file provided by the user. The result of the parsing phase is an instance of the BasicScene class, which stores the scene specification, but not in a form yet suitable for rendering. In the second phase of execution, pbrt creates specific objects corresponding to the scene. In the third phase, the main rendering loop executes. This phase is where pbrt usually spends the majority of its running time. To orchestrate the rendering, pbrt implements an *integrator*, so-named because its main task is to evaluate the integral in Equation (1.1).

> ### 1.3.1 执行阶段
>
> pbrt 在概念上可以分为三个执行阶段。首先，它解析用户提供的场景描述文件。解析阶段的结果是 BasicScene 类的实例，它存储了场景规范，但尚未以适合渲染的形式。在第二个执行阶段，pbrt 创建与场景对应的具体对象。在第三个阶段，主渲染循环执行。这个阶段通常占据 pbrt 大部分的运行时间。为了编排渲染，pbrt 实现了一个*积分器*（integrator），之所以如此命名，是因为它的主要任务是计算方程（1.1）中的积分。

### 1.3.2 pbrt's main() Function

The main() function for the pbrt executable is defined in the file cmd/pbrt.cpp. It is only a hundred and fifty or so lines of code, much of it devoted to processing command-line arguments and related bookkeeping.

> ### 1.3.2 pbrt 的 main() 函数
>
> pbrt 可执行文件的 main() 函数定义在文件 cmd/pbrt.cpp 中。它只有大约一百五十行代码，其中大部分用于处理命令行参数和相关的簿记工作。

The main steps are:
1. Convert command-line arguments to vector of strings
2. Declare variables for parsed command line
3. Process command-line arguments
4. Initialize pbrt via `InitPBRT(options)`
5. Parse provided scene description files via `ParseFiles(&builder, filenames)`
6. Render the scene
7. Clean up via `CleanupPBRT()`

> 主要步骤是：
> 1. 将命令行参数转换为字符串向量
> 2. 声明解析后的命令行变量
> 3. 处理命令行参数
> 4. 通过 `InitPBRT(options)` 初始化 pbrt
> 5. 通过 `ParseFiles(&builder, filenames)` 解析提供的场景描述文件
> 6. 渲染场景
> 7. 通过 `CleanupPBRT()` 清理

After the arguments have been parsed and validated, the ParseFiles() function handles the first of the three phases of execution. After the scene description has been parsed, one of two functions is called to render the scene. RenderWavefront() supports both the CPU and GPU rendering paths, processing a million or so image samples in parallel. RenderCPU() renders the scene using an Integrator implementation and is only available when running on the CPU.

> 参数解析和验证完成后，ParseFiles() 函数处理三个执行阶段中的第一个。场景描述解析完成后，调用两个函数之一来渲染场景。RenderWavefront() 支持 CPU 和 GPU 两种渲染路径，并行处理大约一百万个图像样本。RenderCPU() 使用 Integrator 实现渲染场景，仅在 CPU 上运行时可用。

### 1.3.3 Integrator Interface

In the RenderCPU() rendering path, an instance of a class that implements the Integrator interface is responsible for rendering. Because Integrator implementations only run on the CPU, we will define Integrator as a standard base class with pure virtual methods.

> ### 1.3.3 积分器接口
>
> 在 RenderCPU() 渲染路径中，实现 Integrator 接口的类的实例负责渲染。因为 Integrator 的实现仅在 CPU 上运行，我们将 Integrator 定义为具有纯虚方法的标准基类。

The base Integrator constructor takes a single Primitive that represents all the geometric objects in the scene as well as an array that holds all the lights in the scene. Each geometric object in the scene is represented by a Primitive, which is primarily responsible for combining a Shape that specifies its geometry and a Material that describes its appearance. All the geometric primitives in a scene are collected into a single aggregate primitive stored in the Integrator::aggregate member variable. This aggregate is a special kind of primitive that itself holds references to many other primitives, stored in an acceleration data structure that reduces the number of unnecessary ray intersection tests.

> Integrator 的基类构造函数接受一个代表场景中所有几何对象的 Primitive，以及一个持有场景中所有光源的数组。场景中的每个几何对象由一个 Primitive 表示，它主要负责将指定几何形状的 Shape 和描述外观的 Material 组合在一起。场景中的所有几何图元被收集到一个单一的聚合图元中，存储在 Integrator::aggregate 成员变量中。这个聚合体是一种特殊的图元，它自身持有对许多其他图元的引用，存储在一个减少不必要光线求交测试次数的加速数据结构中。

Each light source in the scene is represented by an object that implements the Light interface. Some lights need to know the bounding box of the entire scene. Therefore, the Integrator constructor calls their Preprocess() methods, providing those bounds. At this point any "infinite" lights are also stored in a separate array.

> 场景中的每个光源由一个实现 Light 接口的对象表示。一些光源需要知道整个场景的包围盒。因此，Integrator 构造函数调用它们的 Preprocess() 方法，提供这些边界。此时任何"无限远"光源也会被存储在一个单独的数组中。

Integrators must provide an implementation of the Render() method, which takes no further arguments. The task of integrators is to render the scene as specified by the aggregate and the lights.

> 积分器必须提供 Render() 方法的实现，该方法不接受其他参数。积分器的任务是按照聚合体和光源的规定渲染场景。

The Integrator class also provides Intersect() and IntersectP() methods for ray–primitive intersection. Intersect() traces the given ray into the scene and returns the closest intersection. IntersectP() only returns a Boolean indicating whether an intersection was found and is generally more efficient—used for shadow rays.

> Integrator 类还提供了用于光线–图元求交的 Intersect() 和 IntersectP() 方法。Intersect() 将给定光线追踪到场景中并返回最近的交点。IntersectP() 只返回一个布尔值表示是否找到交点，通常更高效——用于阴影光线。

Note the capitalized floating-point type Float: almost all floating-point values in pbrt are declared as Floats. Depending on the compilation flags, Float is an alias for either float or double, though single precision float is almost always sufficient in practice.

> 注意大写的浮点类型 Float：pbrt 中几乎所有浮点值都声明为 Float。根据编译标志，Float 是 float 或 double 的别名，尽管单精度 float 在实践中几乎总是足够的。

### 1.3.4 ImageTileIntegrator and the Main Rendering Loop

All of pbrt's CPU-based integrators render images using a camera model to define the viewing parameters, and all parallelize rendering by splitting the image into tiles and having different processors work on different tiles. Therefore, pbrt includes an ImageTileIntegrator that provides common functionality for those tasks.

> ### 1.3.4 ImageTileIntegrator 和主渲染循环
>
> pbrt 所有基于 CPU 的积分器都使用相机模型定义观察参数来渲染图像，并且都通过将图像分割为瓦片（tile）让不同处理器处理不同瓦片来并行化渲染。因此，pbrt 包含了一个 ImageTileIntegrator，为这些任务提供通用功能。

In addition to the aggregate and the lights, the ImageTileIntegrator constructor takes a Camera that specifies the viewing and lens parameters (position, orientation, focus, field of view), and a Sampler whose role is to choose the points on the image plane and supply random sample values used by integrators for estimating the value of the light transport integral.

> 除了聚合体和光源外，ImageTileIntegrator 构造函数还接受一个指定观察和镜头参数（位置、方向、焦点、视场）的 Camera，以及一个 Sampler，其作用是选择图像平面上的点并提供积分器用于估计光传输积分值的随机样本值。

ImageTileIntegrator::Render() renders the image in *waves* of a few samples per pixel. For the first two waves, only a single sample is taken in each pixel. In the next wave, two samples are taken, with the number of samples doubling after each wave up to a limit. This organization means it is possible to see previews of the final image during rendering where all pixels have some samples.

> ImageTileIntegrator::Render() 以每像素若干样本的*波次*来渲染图像。前两波中，每像素只取一个样本。下一波中取两个样本，每波后样本数翻倍直到上限。这种组织方式意味着在渲染期间可以看到最终图像的预览，所有像素都有一些样本。

The ParallelFor2D() function loops over image tiles, running multiple loop iterations concurrently. It automatically chooses a tile size to balance between having significantly more tiles than processors (to avoid idle time) and not having too many tiles (to avoid overhead).

> ParallelFor2D() 函数遍历图像瓦片，并发运行多个循环迭代。它自动选择瓦片大小，以在拥有明显多于处理器数量的瓦片（以避免空闲时间）和不拥有过多瓦片（以避免开销）之间取得平衡。

For each pixel, the thread's Sampler is notified via StartPixelSample(), the integrator's EvaluatePixelSample() method computes the sample's value, and temporary memory in the ScratchBuffer is freed.

> 对于每个像素，通过 StartPixelSample() 通知线程的 Sampler，积分器的 EvaluatePixelSample() 方法计算样本值，并释放 ScratchBuffer 中的临时内存。

### 1.3.5 RayIntegrator Implementation

Just as ImageTileIntegrator centralizes functionality related to integrators that decompose the image into tiles, RayIntegrator provides commonly used functionality to integrators that trace ray paths starting from the camera. All of the integrators implemented in Chapters 13 and 14 inherit from RayIntegrator.

> ### 1.3.5 RayIntegrator 实现
>
> 正如 ImageTileIntegrator 集中了与将图像分解为瓦片相关的积分器功能一样，RayIntegrator 为从相机出发追踪光线路径的积分器提供了常用功能。第 13 和第 14 章中实现的所有积分器都继承自 RayIntegrator。

RayIntegrator implements EvaluatePixelSample(). At the given pixel, it uses its Camera and Sampler to generate a ray into the scene and then calls the Li() method (provided by the subclass) to determine the amount of light arriving at the image plane along that ray. The units of the value returned by Li() are related to the incident spectral radiance at the ray origin, denoted by the symbol Lᵢ in equations—thus, the method name. This value is passed to the Film, which records the ray's contribution to the image.

> RayIntegrator 实现了 EvaluatePixelSample()。在给定像素处，它使用其 Camera 和 Sampler 生成一条射入场景的光线，然后调用 Li() 方法（由子类提供）来确定沿该光线到达图像平面的光量。Li() 返回值的单位与光线原点处的入射光谱辐射有关，在方程中用符号 Lᵢ 表示——因此得名。该值传递给 Film，由 Film 记录光线对图像的贡献。

Each ray carries radiance at a number of discrete wavelengths (four, by default). When computing the color at each pixel, pbrt chooses different wavelengths at different pixel samples so that the final result better reflects the correct result over all wavelengths.

> 每条光线在若干离散波长（默认四个）上携带辐射。在计算每个像素的颜色时，pbrt 在不同的像素样本处选择不同的波长，使最终结果更好地反映所有波长上的正确结果。

CameraRayDifferential also carries a weight associated with the ray that is used to scale the returned radiance value. For simple camera models, each ray is weighted equally, but more accurate camera models might generate rays that contribute differently. Such a camera model might simulate *vignetting*.

> CameraRayDifferential 还携带与光线关联的权重，用于缩放返回的辐射值。对于简单的相机模型，每条光线的权重相同，但更精确的相机模型可能生成贡献不同的光线。这样的相机模型可能模拟*渐晕*效果。

Li() is a pure virtual method that RayIntegrator subclasses must implement. It returns the incident radiance at the origin of a given ray, sampled at the specified wavelengths.

> Li() 是 RayIntegrator 子类必须实现的纯虚方法。它返回给定光线原点处的入射辐射，在指定波长处采样。

### 1.3.6 Random Walk Integrator

The RandomWalkIntegrator implements a simple Monte Carlo approach based on incrementally constructing a *random walk*, where a series of points on scene surfaces are randomly chosen in succession to construct light-carrying paths starting from the camera. This approach models image formation in reverse, starting from the camera rather than from the light sources. Going backward is still physically valid because the physical models of light that pbrt is based on are time-reversible.

> ### 1.3.6 随机游走积分器
>
> RandomWalkIntegrator 实现了一种简单的蒙特卡洛方法，基于增量构建*随机游走*，其中场景表面上的一系列点被依次随机选择，以构建从相机出发的携带光能的路径。这种方法以相反的方式模拟图像形成，从相机而非光源开始。反向进行在物理上仍然有效，因为 pbrt 所基于的光的物理模型是时间可逆的。

Although the implementation of the random walk sampling algorithm is in total just over twenty lines of code, it is capable of simulating complex lighting and shading effects.

> 尽管随机游走采样算法的实现总共只有二十多行代码，但它能够模拟复杂的光照和着色效果。

The algorithm works as follows:

1. **Find intersection**: Trace the ray and find the closest intersection with scene geometry. If no intersection, return emitted light from infinite light sources.

2. **Get emitted radiance**: Evaluate the emitted radiance Le at the intersection point.

3. **Check termination**: If the maximum recursion depth has been reached, return Le only.

4. **Compute BSDF**: Get the BSDF at the intersection point via `isect.GetBSDF()`.

5. **Sample random direction**: Use `SampleUniformSphere(u)` to choose a uniformly distributed random direction.

6. **Evaluate BSDF**: Compute `fcos = bsdf.f(wo, wp) * AbsDot(wp, n)`. If zero, return Le (no contribution).

7. **Recurse**: Spawn a new ray in the sampled direction and recursively call LiRandomWalk(), returning `Le + fcos * LiRandomWalk(...) / (1/(4π))`.

> 算法工作流程如下：
>
> 1. **寻找交点**：追踪光线并找到与场景几何体最近的交点。如果没有交点，返回来自无限远光源的发射光。
>
> 2. **获取发射辐射**：计算交点处的发射辐射 Le。
>
> 3. **检查终止**：如果已达到最大递归深度，仅返回 Le。
>
> 4. **计算 BSDF**：通过 `isect.GetBSDF()` 获取交点处的 BSDF。
>
> 5. **随机采样方向**：使用 `SampleUniformSphere(u)` 选择均匀分布的随机方向。
>
> 6. **评估 BSDF**：计算 `fcos = bsdf.f(wo, wp) * AbsDot(wp, n)`。如果为零则返回 Le（无贡献）。
>
> 7. **递归**：在采样方向上生成新光线并递归调用 LiRandomWalk()，返回 `Le + fcos * LiRandomWalk(...) / (1/(4π))`。

This simple approach has many shortcomings. For example, if the emissive surfaces are small, most ray paths will not find any light and many rays will need to be traced to form an accurate image. In the limit case of a point light source, the image will be black, since there is zero probability of intersecting such a light source. Similar issues apply with BSDF models that scatter light in a concentrated set of directions—a perfect mirror will never be correctly sampled.

> 这种简单方法有许多缺点。例如，如果发光表面很小，大多数光线路径不会找到任何光源，需要追踪许多光线才能形成准确的图像。在点光源的极限情况下，图像将是黑色的，因为与这样的光源相交的概率为零。对于在集中方向上散射光的 BSDF 模型也存在类似问题——完美的镜面反射永远不会被正确采样。

Those issues and more can be addressed through more sophisticated application of Monte Carlo integration techniques. The integrators defined in Chapters 13 through 15 are the culmination of those developments. All still build on the same basic ideas used in the RandomWalkIntegrator, but are much more efficient and robust.

> 这些问题以及更多问题可以通过更精密地应用蒙特卡洛积分技术来解决。第 13 至第 15 章中定义的积分器是这些发展的顶峰。它们仍然建立在 RandomWalkIntegrator 使用的相同基本思想之上，但效率和鲁棒性要高得多。
