## 1.5 Using and Understanding the Code

The pbrt source code distribution is available from pbrt.org. The website also includes additional documentation, images rendered with pbrt, example scenes, errata, and links to a bug reporting system. We encourage you to visit the website and subscribe to the pbrt mailing list.

> pbrt 的源代码发行版可从 pbrt.org 获取。该网站还包括额外的文档、使用 pbrt 渲染的图像、示例场景、勘误表以及错误报告系统的链接。我们鼓励你访问该网站并订阅 pbrt 邮件列表。

pbrt is written in C++, but we have tried to make it accessible to non-C++ experts by limiting the use of esoteric features of the language. Staying close to the core language features also helps with the system's portability. We make use of C++'s extensive standard library whenever it is applicable but will not discuss the semantics of calls to standard library functions in the text. Our expectation is that the reader will consult documentation of the standard library as necessary.

> pbrt 使用 C++ 编写，但我们通过限制使用语言的晦涩特性来尽量让非 C++ 专家也能理解。贴近核心语言特性也有助于系统的可移植性。我们在适用时会使用 C++ 丰富的标准库，但不会在正文中讨论标准库函数调用的语义。我们期望读者根据需要查阅标准库文档。

We will occasionally omit short sections of pbrt's source code from the book. For example, when there are a number of cases to be handled, all with nearly identical code, we will present one case and note that the code for the remaining cases has been omitted from the text. Default class constructors are generally not shown, and the text also does not include details like the various #include directives at the start of each source file. All the omitted code can be found in the pbrt source code distribution.

> 我们偶尔会从书中省略 pbrt 源代码的短小片段。例如，当有多个需要处理的情况且代码几乎相同时，我们会展示一个案例并注明其余情况的代码已从正文中省略。默认类构造函数一般不会展示，正文也不包括每个源文件开头的各种 #include 指令等细节。所有省略的代码都可以在 pbrt 源代码发行版中找到。

### 1.5.1 Source Code Organization

The source code used for building pbrt is under the src directory in the pbrt distribution. In that directory are src/ext, which has the source code for various third-party libraries that are used by pbrt, and src/pbrt, which contains pbrt's source code. We will not discuss the third-party libraries' implementations in the book.

> ### 1.5.1 源代码组织
>
> 用于构建 pbrt 的源代码位于 pbrt 发行版的 src 目录下。该目录中有 src/ext（包含 pbrt 使用的各种第三方库的源代码）和 src/pbrt（包含 pbrt 的源代码）。我们不会在书中讨论第三方库的实现。

The source files in the src/pbrt directory mostly consist of implementations of the various interface types. For example, shapes.h and shapes.cpp have implementations of the Shape interface, materials.h and materials.cpp have materials, and so forth. That directory also holds the source code for parsing pbrt's scene description files.

> src/pbrt 目录中的源文件主要由各种接口类型的实现组成。例如，shapes.h 和 shapes.cpp 包含 Shape 接口的实现，materials.h 和 materials.cpp 包含材质，依此类推。该目录还包含解析 pbrt 场景描述文件的源代码。

The pbrt.h header file in src/pbrt is the first file that is included by all other source files in the system. It contains a few macros and widely useful forward declarations, though we have tried to keep it short and to minimize the number of other headers that it includes in the interests of compile time efficiency.

> src/pbrt 中的 pbrt.h 头文件是系统中所有其他源文件首先包含的文件。它包含一些宏和广泛使用的前向声明，不过为了编译时间效率，我们尽量保持它简短并最小化它包含的其他头文件数量。

The src/pbrt directory also contains a number of subdirectories. They have the following roles:

> src/pbrt 目录还包含若干子目录，它们的作用如下：

- base: Header files defining the interfaces for 12 of the common interface types listed in Table 1.1 (Primitive and Integrator are CPU-only and so are defined in files in the cpu directory).
- cmd: Source files containing the main() functions for the executables that are built for pbrt.
- cpu: CPU-specific code, including Integrator implementations.
- gpu: GPU-specific source code, including functions for allocating memory and launching work on the GPU.
- util: Lower-level utility code, most of it not specific to rendering.
- wavefront: Implementation of the WavefrontPathIntegrator, which is introduced in Chapter 15.

> - base：定义表 1.1 中列出的 12 种通用接口类型接口的头文件（Primitive 和 Integrator 仅限 CPU，因此定义在 cpu 目录的文件中）。
> - cmd：包含为 pbrt 构建的可执行文件的 main() 函数的源文件。
> - cpu：CPU 专用代码，包括 Integrator 实现。
> - gpu：GPU 专用源代码，包括在 GPU 上分配内存和启动工作的函数。
> - util：较低层的实用代码，大部分与渲染无关。
> - wavefront：WavefrontPathIntegrator 的实现，在第 15 章中介绍。

### 1.5.2 Naming Conventions

Functions and classes are generally named using Camel case, with the first letter of each word capitalized and no delineation for spaces. One exception is some methods of container classes, which follow the naming convention of the C++ standard library when they have matching functionality (e.g., size() and begin() and end() for iterators). Variables also use Camel case, though with the first letter lowercase, except for a few global variables.

> ### 1.5.2 命名约定
>
> 函数和类通常使用驼峰命名法（Camel case），每个单词的首字母大写，不使用空格分隔。一个例外是容器类的某些方法，当它们具有与 C++ 标准库匹配的功能时，会遵循标准库的命名约定（例如，size()、begin() 和 end() 用于迭代器）。变量也使用驼峰命名法，但首字母小写，少数全局变量除外。

We also try to match mathematical notation in naming: for example, we use variables like p for points and w for directions. We will occasionally add a p to the end of a variable to denote a primed symbol: wp for ω'. Underscores are used to indicate subscripts in equations: theta_o for θₒ, for example.

> 我们还尽量在命名中匹配数学符号：例如，我们使用 p 表示点，w 表示方向。我们偶尔会在变量末尾添加 p 来表示带撇的符号：wp 表示 ω'。下划线用于表示方程中的下标：例如 theta_o 表示 θₒ。

Our use of underscores is not perfectly consistent, however. Short variable names often omit the underscore—we use wi for ωᵢ and we have already seen the use of Li for Lᵢ. We also occasionally use an underscore to separate a word from a lowercase mathematical symbol. For example, we use Sample_f for a method that samples a function f rather than Samplef, which would be more difficult to read, or SampleF, which would obscure the connection to the function f.

> 然而，我们对下划线的使用并不完全一致。短变量名通常省略下划线——我们使用 wi 表示 ωᵢ，前面已经看到使用 Li 表示 Lᵢ。我们偶尔也用下划线来将单词与小写数学符号分开。例如，我们用 Sample_f 表示采样函数 f 的方法，而不是更难阅读的 Samplef，或者会模糊与函数 f 之间关联的 SampleF。

### 1.5.3 Pointer or Reference?

C++ provides two different mechanisms for passing an object to a function or method by reference: pointers and references. If a function argument is not intended as an output variable, either can be used to save the expense of passing the entire structure on the stack. The convention in pbrt is to use a pointer when the argument will be completely changed by the function or method, a reference when some of its internal state will be changed but it will not be fully reinitialized, and const references when it will not be changed at all. One important exception to this rule is that we will always use a pointer when we want to be able to pass nullptr to indicate that a parameter is not available or should not be used.

> ### 1.5.3 指针还是引用？
>
> C++ 提供两种不同的机制通过引用将对象传递给函数或方法：指针和引用。如果函数参数不打算作为输出变量，两者都可以用来节省在栈上传递整个结构的开销。pbrt 中的约定是：当参数将被函数或方法完全改变时使用指针，当其部分内部状态会被改变但不会完全重新初始化时使用引用，当不会被改变时使用 const 引用。这条规则的一个重要例外是：当我们希望能够传递 nullptr 来表示某个参数不可用或不应使用时，我们总是使用指针。

### 1.5.4 Abstraction versus Efficiency

One of the primary tensions when designing interfaces for software systems is making a reasonable trade-off between abstraction and efficiency. For example, many programmers religiously make all data in all classes private and provide methods to obtain or modify the values of the data items. For simple classes (e.g., Vector3f), we believe that approach needlessly hides a basic property of the implementation—that the class holds three floating-point coordinates—that we can reasonably expect to never change.

> ### 1.5.4 抽象与效率
>
> 设计软件系统接口时的主要矛盾之一是在抽象和效率之间做出合理的权衡。例如，许多程序员虔诚地将所有类中的所有数据设为私有，并提供方法来获取或修改数据项的值。对于简单的类（如 Vector3f），我们认为这种做法不必要地隐藏了实现的基本属性——该类持有三个浮点坐标——而这一点我们可以合理地预期永远不会改变。

Of course, using no information hiding and exposing all details of all classes' internals leads to a code maintenance nightmare, but we believe that there is nothing wrong with judiciously exposing basic design decisions throughout the system. For example, the fact that a Ray is represented with a point, a vector, a time, and the medium it is in is a decision that does not need to be hidden behind a layer of abstraction. Code elsewhere is shorter and easier to understand when details like these are exposed.

> 当然，完全不使用信息隐藏而暴露所有类内部的所有细节会导致代码维护的噩梦，但我们认为在整个系统中审慎地暴露基本设计决策并无不妥。例如，Ray 由一个点、一个向量、一个时间和它所处的介质表示这一事实，不需要隐藏在一层抽象之后。当这些细节被暴露时，其他地方的代码会更简短、更容易理解。

An important thing to keep in mind when writing a software system and making these sorts of trade-offs is the expected final size of the system. pbrt is roughly 70,000 lines of code and it is never going to grow to be a million lines of code; this fact should be reflected in the amount of information hiding used in the system.

> 在编写软件系统和做出这类权衡时需要牢记的一个重要事项是系统的预期最终规模。pbrt 大约有 70,000 行代码，它永远不会增长到一百万行代码；这一事实应该反映在系统中使用的信息隐藏程度上。

### 1.5.5 pstd

We have reimplemented a subset of the C++ standard library in the pstd namespace; this was necessary in order to use those parts of it interchangeably on the CPU and on the GPU. For the purposes of reading pbrt's source code, anything in pstd provides the same functionality with the same type and methods as the corresponding entity in std. We will therefore not document usage of pstd in the text here.

> ### 1.5.5 pstd
>
> 我们在 pstd 命名空间中重新实现了 C++ 标准库的一个子集；这是为了能在 CPU 和 GPU 上互换使用这些部分所必需的。就阅读 pbrt 源代码而言，pstd 中的任何内容都提供与 std 中对应实体相同的功能、相同的类型和方法。因此我们不会在此正文中记录 pstd 的用法。

### 1.5.6 Allocators

Almost all dynamic memory allocation for the objects that represent the scene in pbrt is performed using an instance of an Allocator that is provided to the object creation methods. In pbrt, Allocator is shorthand for the C++ standard library's pmr::polymorphic_allocator type.

> ### 1.5.6 分配器
>
> 在 pbrt 中，表示场景的对象几乎所有的动态内存分配都是使用提供给对象创建方法的 Allocator 实例来执行的。在 pbrt 中，Allocator 是 C++ 标准库 pmr::polymorphic_allocator 类型的简写。

```cpp
using Allocator = pstd::pmr::polymorphic_allocator<std::byte>;
```

std::pmr::polymorphic_allocator implementations provide a few methods for allocating and freeing objects. These three are used widely in pbrt:

> std::pmr::polymorphic_allocator 的实现提供了几种分配和释放对象的方法。以下三种在 pbrt 中广泛使用：

- `allocate_bytes(size_t nbytes, size_t alignment)` — allocates the specified number of bytes of memory.
- `allocate_object<T>(size_t n = 1)` — allocates an array of n objects of type T, initializing each with its default constructor.
- `new_object<T, Args>(Args&&... args)` — allocates a single object of type T and calls its constructor with the provided arguments.

> - `allocate_bytes(size_t nbytes, size_t alignment)` — 分配指定字节数的内存。
> - `allocate_object<T>(size_t n = 1)` — 分配 n 个类型为 T 的对象数组，每个对象用默认构造函数初始化。
> - `new_object<T, Args>(Args&&... args)` — 分配一个类型为 T 的对象，并用提供的参数调用其构造函数。

There are corresponding methods for freeing each type of allocation: deallocate_bytes(), deallocate_object(), and delete_object().

> 有对应的方法用于释放每种类型的分配：deallocate_bytes()、deallocate_object() 和 delete_object()。

A tricky detail related to the use of allocators with data structures from the C++ standard library is that a container's allocator is fixed once its constructor has run. Thus, if one container is assigned to another, the target container's allocator is unchanged even though all the values it stores are updated. (This is the case even with C++'s move semantics.) Therefore, it is common to see objects' constructors in pbrt passing along an allocator in member initializer lists for containers that they store even if they are not yet ready to set the values stored in them.

> 使用分配器与 C++ 标准库数据结构相关的一个棘手细节是：容器的分配器在构造函数运行后就固定了。因此，如果将一个容器赋值给另一个容器，目标容器的分配器不会改变，即使它存储的所有值都被更新了。（即使使用 C++ 的移动语义也是如此。）因此，在 pbrt 中常见对象的构造函数在成员初始化列表中为它们存储的容器传递分配器，即使它们还没有准备好设置存储在其中的值。

Using an explicit memory allocator rather than direct calls to new and delete has a few advantages. Not only does it make it easy to do things like track the total amount of memory that has been allocated, but it also makes it easy to substitute allocators that are optimized for many small allocations, as is useful when building acceleration structures in Chapter 7. Using allocators in this way also makes it easy to store the scene objects in memory that is visible to the GPU when GPU rendering is being used.

> 使用显式内存分配器而非直接调用 new 和 delete 有几个优势。它不仅使跟踪已分配内存总量等操作变得容易，还使替换为针对大量小型分配优化的分配器变得简单，这在第 7 章构建加速结构时很有用。以这种方式使用分配器还使得在使用 GPU 渲染时，可以轻松地将场景对象存储在对 GPU 可见的内存中。

### 1.5.7 Dynamic Dispatch

As mentioned in Section 1.3, virtual functions are generally not used for dynamic dispatch with polymorphic types in pbrt (the main exception being the Integrators). Instead, the TaggedPointer class is used to represent a pointer to one of a specified set of types; it includes machinery for runtime type identification and thence dynamic dispatch. Two considerations motivate its use.

> ### 1.5.7 动态分派
>
> 如 1.3 节所述，pbrt 中多态类型的动态分派通常不使用虚函数（主要的例外是 Integrator）。取而代之的是使用 TaggedPointer 类来表示指向一组指定类型之一的指针；它包含用于运行时类型识别和动态分派的机制。两个考虑因素促使了它的使用。

First, in C++, an instance of an object that inherits from an abstract base class includes a hidden virtual function table pointer that is used to resolve virtual function calls. On most modern systems, this pointer uses eight bytes of memory. While eight bytes may not seem like much, we have found that when rendering complex scenes with previous versions of pbrt, a substantial amount of memory would be used just for virtual function pointers for shapes and primitives. With the TaggedPointer class, there is no incremental storage cost for type information.

> 首先，在 C++ 中，继承自抽象基类的对象实例包含一个隐藏的虚函数表指针，用于解析虚函数调用。在大多数现代系统上，这个指针占用 8 字节内存。虽然 8 字节似乎不多，但我们发现在使用 pbrt 以前的版本渲染复杂场景时，仅形状和图元的虚函数指针就会占用大量内存。使用 TaggedPointer 类，类型信息没有额外的存储开销。

The other problem with virtual function tables is that they store function pointers that point to executable code. This characteristic means that a virtual function table can be valid for method calls from either the CPU or from the GPU, but not from both simultaneously, since the executable code for the different processors is stored at different memory locations. When using the GPU for rendering, it is useful to be able to call methods from both processors, however.

> 虚函数表的另一个问题是它们存储的是指向可执行代码的函数指针。这一特性意味着虚函数表只能对 CPU 或 GPU 的方法调用有效，不能同时对两者有效，因为不同处理器的可执行代码存储在不同的内存位置。然而，当使用 GPU 进行渲染时，能够从两个处理器调用方法是有用的。

For all the code that just calls methods of polymorphic objects, the use of pbrt's TaggedPointer in place of virtual functions makes no difference other than the fact that method calls are made using the . operator, just as would be used for a C++ reference.

> 对于所有只调用多态对象方法的代码，使用 pbrt 的 TaggedPointer 代替虚函数没有任何区别，只是方法调用使用 . 运算符，就像用于 C++ 引用一样。

### 1.5.8 Code Optimization

We have tried to make pbrt efficient through the use of well-chosen algorithms rather than through local micro-optimizations, so that the system can be more easily understood. However, efficiency is an integral part of rendering, and so we discuss performance issues throughout the book.

> ### 1.5.8 代码优化
>
> 我们努力通过使用精心选择的算法而非局部微优化来使 pbrt 高效，以便系统更容易被理解。然而，效率是渲染不可或缺的一部分，因此我们在全书中都讨论性能问题。

For both CPUs and GPUs, processing performance continues to grow more quickly than the speed at which data can be loaded from main memory into the processor. This means that waiting for values to be fetched from memory can be a major performance limitation. The most important optimizations that we discuss relate to minimizing unnecessary memory access and organizing algorithms and data structures in ways that lead to coherent access patterns.

> 对于 CPU 和 GPU 而言，处理性能的增长速度持续快于从主内存加载数据到处理器的速度。这意味着等待从内存获取数据可能是主要的性能瓶颈。我们讨论的最重要的优化与最小化不必要的内存访问以及以产生连贯访问模式的方式组织算法和数据结构有关。

### 1.5.9 Debugging and Logging

Debugging a renderer can be challenging, especially in cases where the result is correct most of the time but not always. pbrt includes a number of facilities to ease debugging.

> ### 1.5.9 调试和日志
>
> 调试渲染器可能很有挑战性，特别是在结果大部分时间正确但不总是正确的情况下。pbrt 包含一些便于调试的设施。

One of the most important is a suite of unit tests. We have found unit testing to be invaluable in the development of pbrt for the reassurance it gives that the tested functionality is very likely to be correct. There are many assertions throughout the pbrt codebase, most of them not included in the book text. These check conditions that should never be true and issue an error and exit immediately if they are found to be true at runtime.

> 最重要的之一是一套单元测试。我们发现单元测试在 pbrt 的开发中是无价的，因为它保证了被测试的功能很可能是正确的。pbrt 代码库中有许多断言，其中大部分没有包含在书中。这些断言检查不应为真的条件，如果在运行时发现为真则立即报错并退出。

We have also endeavored to make the execution of pbrt at a given pixel sample deterministic. One challenge with debugging a renderer is a crash that only happens after minutes or hours of rendering computation. With deterministic execution, rendering can be restarted at a single pixel sample in order to more quickly return to the point of a crash.

> 我们还努力使 pbrt 在给定像素样本上的执行具有确定性。调试渲染器的一个挑战是崩溃只在数分钟或数小时的渲染计算后才发生。通过确定性执行，可以从单个像素样本重新开始渲染，以便更快地返回到崩溃点。

Finally, it is often useful to print out the values stored in a data structure during the course of debugging. We have implemented ToString() methods for nearly all of pbrt's classes. They return a std::string representation of them so that it is easy to print their full object state during program execution.

> 最后，在调试过程中打印数据结构中存储的值通常很有用。我们为几乎所有 pbrt 的类实现了 ToString() 方法。它们返回对象的 std::string 表示，以便在程序执行期间轻松打印其完整的对象状态。

### 1.5.10 Parallelism and Thread Safety

In pbrt (as is the case for most ray tracers), the vast majority of data at rendering time is read only (e.g., the scene description and texture images). Much of the parsing of the scene file and creation of the scene representation in memory is done with a single thread of execution, so there are few synchronization issues during that phase of execution. During rendering, concurrent read access to all the read-only data by multiple threads works with no problems on both the CPU and the GPU; we only need to be concerned with situations where data in memory is being modified.

> ### 1.5.10 并行性和线程安全
>
> 在 pbrt 中（大多数光线追踪器也是如此），渲染时绝大多数数据是只读的（例如场景描述和纹理图像）。场景文件的大部分解析和内存中场景表示的创建都是在单线程中完成的，因此在该执行阶段几乎没有同步问题。在渲染期间，多线程对所有只读数据的并发读取在 CPU 和 GPU 上都没有问题；我们只需要关注内存中数据被修改的情况。

As a general rule, the low-level classes and structures in the system are not thread-safe. For example, the Point3f class, which stores three float values to represent a point in 3D space, is not safe for multiple threads to call methods that modify it at the same time. The runtime overhead to make Point3f thread-safe would have a substantial effect on performance with little benefit in return.

> 一般来说，系统中的低层类和结构不是线程安全的。例如，存储三个 float 值来表示三维空间中一个点的 Point3f 类，对于多线程同时调用修改它的方法是不安全的。使 Point3f 线程安全的运行时开销会对性能产生显著影响而回报甚微。

The same is true for classes like Vector3f, Normal3f, SampledSpectrum, Transform, Quaternion, and SurfaceInteraction. These classes are usually either created at scene construction time and then used as read-only data or allocated on the stack during rendering and used only by a single thread.

> Vector3f、Normal3f、SampledSpectrum、Transform、Quaternion 和 SurfaceInteraction 等类也是如此。这些类通常在场景构建时创建然后作为只读数据使用，或者在渲染期间在栈上分配并仅由单个线程使用。

With two exceptions, implementations of the base types listed in Table 1.1 are safe for multiple threads to use simultaneously. The first exceptions are the Light Preprocess() method implementations. The second exception is Sampler class implementations; their methods are also not expected to be thread-safe. Therefore, a unique Sampler is created for each rendering thread using Sampler::Clone().

> 除了两个例外，表 1.1 中列出的基本类型的实现对于多线程同时使用是安全的。第一个例外是 Light 的 Preprocess() 方法实现。第二个例外是 Sampler 类的实现；它们的方法也不期望是线程安全的。因此，使用 Sampler::Clone() 为每个渲染线程创建唯一的 Sampler。

All stand-alone functions in pbrt are thread-safe (as long as multiple threads do not pass pointers to the same data to them).

> pbrt 中所有独立函数都是线程安全的（只要多个线程不向它们传递指向相同数据的指针）。

### 1.5.11 Extending the System

One of our goals in writing this book and building the pbrt system was to make it easier for developers and researchers to experiment with new (or old!) ideas in rendering. One of the great joys in computer graphics is writing new software that makes a new image; even small changes to the system can be fun to experiment with.

> ### 1.5.11 扩展系统
>
> 我们编写本书和构建 pbrt 系统的目标之一是让开发者和研究人员更容易地实验渲染中的新（或旧！）想法。计算机图形学的一大乐趣就是编写新软件来生成新图像；即使是对系统的小修改也很有趣。

### 1.5.12 Bugs

Although we made every effort to make pbrt as correct as possible through extensive testing, it is inevitable that some bugs are still present.

> ### 1.5.12 Bug
>
> 尽管我们通过广泛的测试尽一切努力使 pbrt 尽可能正确，但不可避免地仍存在一些 bug。

If you believe you have found a bug in the system, please do the following:

> 如果你认为发现了系统中的 bug，请执行以下操作：

1. Reproduce the bug with an unmodified copy of the latest version of pbrt.
2. Check the online discussion forum and the bug-tracking system at pbrt.org.
3. Try to find the simplest possible test case that demonstrates the bug.
4. Submit a detailed bug report using our online bug-tracking system.

> 1. 使用最新版本的未修改 pbrt 副本复现该 bug。
> 2. 在 pbrt.org 查看在线讨论论坛和 bug 跟踪系统。
> 3. 尝试找到能演示该 bug 的最简单测试用例。
> 4. 使用我们的在线 bug 跟踪系统提交详细的 bug 报告。

We will periodically update the pbrt source code repository with bug fixes and minor enhancements. However, we will not make major changes to the pbrt source code so that it does not diverge from the system described here in the book.

> 我们会定期用 bug 修复和小改进更新 pbrt 源代码仓库。但是，我们不会对 pbrt 源代码进行重大更改，以使其不会偏离本书中描述的系统。
