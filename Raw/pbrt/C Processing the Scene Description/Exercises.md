1. ③ An advantage of the way that pbrt separates parsing, graphics state management, and the creation of scene objects is that it is easier to replace or extend those components of the system than it might be if all those responsibilities were in a single class. Investigate pbrt 's parsing performance with scenes that have multi-gigabyte \*.pbrt scene description files (Disney's *Moana Island* scene ([Walt Disney Animation Studios 2018](https://pbr-book.org/4ed/Processing_the_Scene_Description/Further_Reading.html#cite:DisneyMoana)) is a good choice) and develop a scene description format for pbrt that is more efficient to parse. You might, for example, consider a compact binary format. Take advantage of the [ParserTarget](https://pbr-book.org/4ed/Processing_the_Scene_Description/Tokenizing_and_Parsing.html#ParserTarget) interface to write a converter from pbrt 's current scene file format to your format and then implement new parsing routines that call [ParserTarget](https://pbr-book.org/4ed/Processing_the_Scene_Description/Tokenizing_and_Parsing.html#ParserTarget) interface methods. Use a profiler to measure how much time is spent in parsing before and after your changes. What is the performance benefit from your representation? How much smaller are file sizes?
> ③ pbrt 将解析、图形状态管理和场景对象创建分离的方式的一个优点是，与将所有这些职责放在单个类中相比，更容易替换或扩展系统的这些组件。使用具有数 GB 大小 \*.pbrt 场景描述文件的场景（迪士尼的 *Moana Island* 场景（[Walt Disney Animation Studios 2018](https://pbr-book.org/4ed/Processing_the_Scene_Description/Further_Reading.html#cite:DisneyMoana)）是一个好选择）调查 pbrt 的解析性能，并开发一种更高效的 pbrt 场景描述格式。例如，你可以考虑紧凑的二进制格式。利用 [ParserTarget](https://pbr-book.org/4ed/Processing_the_Scene_Description/Tokenizing_and_Parsing.html#ParserTarget) 接口编写从 pbrt 当前场景文件格式到你格式的转换器，然后实现调用 [ParserTarget](https://pbr-book.org/4ed/Processing_the_Scene_Description/Tokenizing_and_Parsing.html#ParserTarget) 接口方法的新解析例程。使用性能分析器测量修改前后解析所花费的时间。你的表示带来了多少性能优势？文件大小减小了多少？
2. ③ Generalize pbrt 's mechanism for specifying animation; the current implementation only allows the user to provide two transformation matrices, at the start and end of a fixed time range. For specifying more complex motion, a more flexible approach may be useful. One improvement is to allow the user to specify an arbitrary number of *keyframe* transformations, each associated with an arbitrary time. More generally, the system could be extended to support transformations that are explicit functions of time. For example, a rotation could be described with an expression of the form Rotate (time \* 2 + 1) 0 0 1 to describe a time-varying rotation about the axis. Extend pbrt to support a more general matrix animation scheme, and render images showing results that are not possible with the current implementation. Is there a performance cost due to your changes for scenes with animated objects that do not need the generality of your improvements?
> ③ 推广 pbrt 指定动画的机制；当前实现只允许用户在固定时间范围的开始和结束提供两个变换矩阵。为了指定更复杂的运动，更灵活的方法可能很有用。一个改进是允许用户指定任意数量的*关键帧*变换，每个关键帧关联一个任意时间。更一般地，系统可以扩展为支持作为时间显式函数的变换。例如，可以用 Rotate (time \* 2 + 1) 0 0 1 形式的表达式描述绕轴的时变旋转。扩展 pbrt 以支持更通用的矩阵动画方案，并渲染展示当前实现无法实现的结果的图像。对于具有动画对象但不需要你改进的通用性的场景，你的修改是否有性能开销？
3. ③ Extend pbrt to have some retained mode semantics so that animated sequences of images can be rendered without needing to respecify the entire scene for each frame. Make sure that it is possible to remove some objects from the scene, add others, modify objects' materials and transformations from frame to frame, and so on. Measure the performance benefit from your approach versus the current implementation. How is the benefit affected by how fast rendering is?
> ③ 扩展 pbrt 使其具有一些保留模式语义，以便可以渲染动画图像序列而无需为每一帧重新指定整个场景。确保可以从场景中移除某些对象、添加其他对象、逐帧修改对象的材质和变换等。测量你的方法相对于当前实现的性能优势。渲染速度对这种优势有什么影响？
4. ② In pbrt 's current implementation, a unique [TransformedPrimitive](https://pbr-book.org/4ed/Primitives_and_Intersection_Acceleration/Primitive_Interface_and_Geometric_Primitives.html#TransformedPrimitive) is created for each [Shape](https://pbr-book.org/4ed/Shapes/Basic_Shape_Interface.html#Shape) with an animated transformation when the CPU is used for rendering. If many shapes have exactly the same animated transformation, this turns out to be a poor choice. Consider the difference between a million-triangle mesh with an animated transformation versus a million independent triangles, all of which happen to have the same animated transformation. In the first case, all the triangles in the mesh are stored in a single instance of a [TransformedPrimitive](https://pbr-book.org/4ed/Primitives_and_Intersection_Acceleration/Primitive_Interface_and_Geometric_Primitives.html#TransformedPrimitive) with an animated transformation. If a ray intersects the bounding box that encompasses all the object's motion over the frame time, then it is transformed to the mesh's object space according to the interpolated transformation at the ray's time. At this point, the intersection computation is no different from the intersection test with a static primitive; the only overhead due to the animation is from the larger bounding box and rays that hit the bounding box but not the animated primitive and the extra computation for matrix interpolation and transforming each ray once, according to its time. In the second case, each triangle is stored in its own [TransformedPrimitive](https://pbr-book.org/4ed/Primitives_and_Intersection_Acceleration/Primitive_Interface_and_Geometric_Primitives.html#TransformedPrimitive), all of which happen to have the same AnimatedTransform. Each instance of [TransformedPrimitive](https://pbr-book.org/4ed/Primitives_and_Intersection_Acceleration/Primitive_Interface_and_Geometric_Primitives.html#TransformedPrimitive) will have a large bounding box to encompass each triangle's motion, giving the acceleration structure a difficult set of inputs to deal with: many primitives with substantially overlapping bounding boxes. The impact on ray–primitive intersection efficiency will be high: the ray will be redundantly transformed many times by what happens to be the same recomputed interpolated transformation, and many intersection tests will be performed due to the large bounding boxes. Performance will be much worse than the first case. To address this case, modify the code that creates primitives so that if independent shapes are provided with the same animated transformation, they are all collected into a single acceleration structure with a single animated transformation. What is the performance improvement for the worst case outlined above? Are there cases where the current implementation is a better choice?
> ② 在 pbrt 的当前实现中，当使用 CPU 渲染时，为每个具有动画变换的 [Shape](https://pbr-book.org/4ed/Shapes/Basic_Shape_Interface.html#Shape) 创建一个唯一的 [TransformedPrimitive](https://pbr-book.org/4ed/Primitives_and_Intersection_Acceleration/Primitive_Interface_and_Geometric_Primitives.html#TransformedPrimitive)。如果许多形状具有完全相同的动画变换，这就是一个糟糕的选择。考虑具有动画变换的百万三角形网格与一百万个独立三角形（碰巧都具有相同的动画变换）之间的区别。在第一种情况下，网格中的所有三角形存储在具有动画变换的单个 [TransformedPrimitive](https://pbr-book.org/4ed/Primitives_and_Intersection_Acceleration/Primitive_Interface_and_Geometric_Primitives.html#TransformedPrimitive) 实例中。如果光线与包围对象在帧时间内所有运动的包围盒相交，则根据光线时间处的插值变换将其变换到网格的对象空间。此时，求交计算与静态图元的求交测试没有区别；由于动画造成的唯一开销来自更大的包围盒和击中包围盒但未击中动画图元的光线，以及根据光线时间对每条光线进行一次矩阵插值和变换的额外计算。在第二种情况下，每个三角形存储在自己的 [TransformedPrimitive](https://pbr-book.org/4ed/Primitives_and_Intersection_Acceleration/Primitive_Interface_and_Geometric_Primitives.html#TransformedPrimitive) 中，碰巧都具有相同的 AnimatedTransform。每个实例都将有一个大的包围盒来包围每个三角形的运动，给加速结构一个难以处理的输入集：许多具有大量重叠包围盒的图元。修改创建图元的代码，以便如果独立形状提供了相同的动画变换，它们都被收集到具有单个动画变换的单个加速结构中。上述最坏情况的性能改善是多少？是否存在当前实现是更好选择的情况？
